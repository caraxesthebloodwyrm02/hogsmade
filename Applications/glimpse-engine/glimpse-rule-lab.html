Glimpse Rule Lab

Engineering playground for authoring, testing, and understanding Glimpse rules.

How the System Works

When you drop a data file into Glimpse, it flows through four stages:

1. Ingest
Parse CSV/JSON
→
2. Profile
Detect fields, types, dimensions
→
3. Rules
Fire rules, collect evidence
→
4. Articulate
Score lenses, rank views, render

Rules are the brain. Each rule says: "IF these conditions are true about the data, THEN boost this context lens or prefer this visual view." Rules produce evidence. Evidence drives everything you see.

Anatomy of a Rule

Every rule has the same shape. Here's what each part means:

The Template

id
Unique name (kebab-case)
label
Human-readable description
applies_to
What does this rule inspect? dataset, entity, or relation
function
Which safe function to call (e.g. taxonomy_score)
args
Arguments passed to the function
guards
Preconditions that must pass first
derive
What happens when the rule fires: boost_lens or prefer_view
because
Plain English explanation of why this rule exists
priority
Higher number = runs first (1-100)
promotion
experimental or active

Example in Plain Language

Rule: "When records share the same country, favor the map view."

Translation:
• applies_to: relation (comparing two records)
• function: shared_dimension (check if they share a value)
• args: dimension = space
• guards: dataset must have a space dimension
• derive: prefer_view = map, score = 0.25
• because: Shared place relations are more legible in spatial views.
Rule: "Education keywords boost the education lens."

Translation:
• applies_to: entity (each record)
• function: taxonomy_score (count keyword matches)
• args: domain = education, min_score = 1
• derive: boost_lens = education, score = 0.7
• because: Learning terms indicate pedagogical framing.

Function Reference

Every function is a safe, validated operation. The engine will not run a function if its arguments are invalid.

Function What it does Arguments Returns Use when...
taxonomy_score Counts keyword matches for a domain path, domain, min_score score You want to detect a topic/domain in the data text
tone_score Counts emotional tone cues path, tone, min_score score Narrative data has moods (anxious, determined, etc.)
semantic_proximity Finds synonym-expanded terms in text path, term, min_matches score You want fuzzy matching — "place" also finds "region", "locale"
shared_dimension Checks if two records share a dimension value dimension boolean You're comparing relationships (same country? same era?)
temporal_distance Measures year gap between two records max_gap score Time proximity matters for the relationship
influence_link Detects explicit influence chains (none) boolean Data has influenced_by or inspired_by fields
data_shape Classifies dataset complexity min_records score You want rules that respond to data size/shape
density_score Measures records x fields ratio dense_threshold score Dense data should use matrix, sparse should use graph
relationship_type Detects hierarchy, network, sequence, etc. (none) score You want the system to auto-detect the data structure
field_pattern Matches field names by regex pattern boolean Specific field names (parent, child, step) signal structure
cardinality_check Counts distinct values in a dimension dimension, max_distinct score Low cardinality = good for clusters. High = needs different encoding
record_range Checks if record count is in a range min, max boolean Small datasets work well with graphs. Large ones need tables.
field_exists Checks if a path has a value path boolean Guard: only run a rule if a certain field exists
equals_value Compares a path to an exact value path, value boolean Guard: only run if a flag is true
numeric_threshold Compares a number against a threshold path, op, value score Score-based gating

Common Recipes

Copy-paste these patterns when you want to add new rules.

Add a New Domain

3 steps:

  1. Add domain to taxonomy.domains in YAML with id, label, keywords
  2. Create a rule using taxonomy_score pointing to your new domain
  3. Add the rule id to rule_sets.base.rules

Add a New View Preference

When data shape should prefer a specific view:

  1. Pick the right function (density_score, field_pattern, record_range)
  2. Set derive: prefer_view
  3. Add a guard if the preference requires a dimension to exist

Add a New Preset

Presets change weights without changing rules:

  1. Add to presets section with label
  2. Set lens_weights (multiply lens scores)
  3. Set view_bias (multiply view scores)

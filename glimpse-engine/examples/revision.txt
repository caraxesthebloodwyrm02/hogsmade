❯ create a ruleset

● Based on the boundaries we extracted:

  --------------------------------------------------------------------------------------------------------------------------------

  CascadeProjects Automation Ruleset

   R1  External data never writes to the repo directly.
       All inbound API/data changes enter through a dedicated generation workflow
       that opens a PR. No direct commits from external sources.

   R2  Style is post-merge, not pre-merge.
       Formatters run after merge to main. They never block a PR.
       Humans do not fix formatting manually.

   R3  AI reviews; humans merge.
       Automated review (Gemini, Copilot, etc.) is a required signal, not an approval.
       No bot may merge a PR. The merge action belongs to a human.

   R4  Bots own version numbers entirely.
       No human edits version fields. Changeset files are the only human input
       into the release process.

   R5  Generation and publishing are separate pipeline stages.
       Code is generated → reviewed → merged before any publish step runs.
       A single workflow must not both generate and publish.

   R6  One bot, one responsibility.
       Each bot identity maps to exactly one pipeline concern.
       Do not reuse a bot token across unrelated workflow jobs.

   R7  All automated commits are attributed.
       Bot commits carry an explicit git identity (name + email).
       No anonymous or inherited-identity commits from automation.

   R8  Concurrency is explicit.
       Any workflow that writes to the repo or opens PRs declares a concurrency group.
       Parallel runs of the same job on the same branch are cancelled or queued,
       never allowed to race.
